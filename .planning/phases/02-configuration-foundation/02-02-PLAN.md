  <action>
    Refactor `Wine.constructWineEnvironment(for:environment:)` in WineEnvironment.swift to:
    1. Create a new `EnvironmentBuilder`
    2. Populate the `base` layer: WINEPREFIX (bottle.url.path), WINEDEBUG ("fixme-all"), GST_DEBUG ("1")
    3. Populate the `platform` layer by calling `Wine.applyMacOSCompatibilityFixes(to:)` -- but instead of mutating a dict, collect the fixes into a temp dict and call `builder.setAll(fixes, layer: .platform)`. The simplest approach: create a temporary `[String: String]`, call the existing `applyMacOSCompatibilityFixes(to: &temp)`, then `builder.setAll(temp, layer: .platform)`. Do NOT modify MacOSCompatibility.swift itself -- just adapt the call pattern.
    4. Populate `bottleManaged`, `launcherManaged` layers by calling new methods on BottleSettings (see below)
    5. Populate `bottleUser` layer: `builder.setAll([:], layer: .bottleUser)` -- the bottle_user layer slot is kept empty for now (no bottle-level custom env vars UI yet, per research open question #3)
    6. Populate `programUser` layer from the `environment` parameter (the caller-provided dict, typically from `Program.generateEnvironment()`)
    7. Leave `featureRuntime` and `callsiteOverride` empty (populated by future phases / direct callers)
    8. Resolve the builder and return the environment dict. Store provenance as a discarded value for now (will be used by Phase 5 diagnostics). Filter out invalid env keys using `Wine.isValidEnvKey()` with debug logging, same as today.

    For the DLL override composition:
    - During layer population, do NOT set WINEDLLOVERRIDES via `builder.set`. Instead, collect DLL override entries from managed sources (DXVK toggle, launcher requiresDXVK) into a `DLLOverrideResolver`, collect bottle custom entries from `bottle.settings.dllOverrides`, and collect program entries from ProgramOverrides (if present in the `environment` param -- see Task 2 for full wiring).
    - After resolve(), render the final WINEDLLOVERRIDES string from `DLLOverrideResolver.resolve()` and set it on the resolved environment dict if non-empty.

    Replace `Wine.constructWineServerEnvironment(for:environment:)` with a call to `constructWineEnvironment` directly (it is currently dead code -- never called from anywhere, per codebase search). Remove the method entirely.

    Refactor `BottleSettings.environmentVariables(wineEnv:)` by splitting it into two new public methods:
    - `populateBottleManagedLayer(builder: inout EnvironmentBuilder)` -- sets all settings-derived env vars (DXVK HUD, DXVK_ASYNC, sync mode, Metal HUD, Metal trace, AVX, DXR, Metal validation, Sequoia compat, performance preset, shader cache, force D3D11) into the `bottleManaged` layer. This replaces the direct dict mutation. The logic is identical to today's `environmentVariables` but calls `builder.set(key, value, layer: .bottleManaged)` instead of `wineEnv.updateValue(value, forKey: key)` and `builder.remove(key, layer: .bottleManaged)` instead of `wineEnv.removeValue(forKey:)`.
    - `populateLauncherManagedLayer(builder: inout EnvironmentBuilder)` -- moves the `applyLauncherCompatibility` logic into the `launcherManaged` layer. Similarly replaces dict mutation with builder calls.
    - `populateInputCompatibilityLayer(builder: inout EnvironmentBuilder)` -- moves `applyInputCompatibility` into `bottleManaged` layer (input settings are bottle-managed toggles).

    IMPORTANT: The DXVK DLL overrides (the `dxgi,d3d9,d3d10core,d3d11=n,b` string) must NOT be set via `builder.set("WINEDLLOVERRIDES", ...)`. Instead, these are collected as structured `DLLOverrideEntry` arrays and fed to the `DLLOverrideResolver`. The `populateBottleManagedLayer` should return (or accept as inout) the managed DLL override entries for DXVK. Similarly, `populateLauncherManagedLayer` should return any launcher-required DLL overrides.

    Keep the `environmentVariables(wineEnv:)` method as a deprecated wrapper that creates a builder, calls the new populators, resolves, and copies back -- this ensures any code not yet migrated still works. Mark it `@available(*, deprecated, message: "Use EnvironmentBuilder layer populators instead")`.

    Key behavioral preservation:
    - macOS 15.4+ ESYNC stability logic: When enhancedSync is .none and macOS >= 15.4, the bottleManaged layer STILL sets WINEESYNC=1 (stability requirement). Provenance will note this.
    - Performance preset logic: Identical to existing `applyPerformancePreset` but through builder.set calls
    - Launcher locale and GPU spoofing: Identical to existing `applyLauncherCompatibility` logic
  </action>

    1. Update `Program.generateEnvironment()` in Program.swift to also produce ProgramOverrides data. The method currently returns `[String: String]` (locale + custom env vars). Change its signature to return a richer type or add a companion method `func resolvedOverrides(from bottle: Bottle) -> ProgramOverrides` that merges `settings.overrides` with bottle defaults (nil fields = use bottle value). Actually, the cleaner approach: don't change generateEnvironment(). Instead, in `Wine.constructWineEnvironment`, accept an optional `ProgramOverrides?` parameter. The overrides are applied in the `programUser` layer:
       - If overrides.dxvk != nil, set WINEDLLOVERRIDES entries accordingly (via DLLOverrideResolver)
       - If overrides.enhancedSync != nil, set WINEESYNC/WINEMSYNC in programUser layer (overrides bottleManaged)
       - If overrides.dxvkAsync != nil, set DXVK_ASYNC in programUser
       - etc. for each overrideable field
       This means `constructWineEnvironment` gets a new parameter: `programOverrides: ProgramOverrides? = nil`

    2. Update `Wine.runProgram(at:args:bottle:environment:)` in Wine.swift to accept a `programOverrides: ProgramOverrides? = nil` parameter and pass it to `constructWineEnvironment`. The public `runWineProcess(name:args:bottle:environment:)` does NOT need programOverrides (it's for arbitrary Wine commands, not program launches).

    3. Update `Program+Extensions.swift` `launchWithUserMode` to pass `settings.overrides` to `Wine.runProgram`.

    4. Update `Wine.generateRunCommand` and `Wine.generateTerminalEnvironmentCommand` to use EnvironmentBuilder (they currently call `constructWineEnvironment` which is already refactored, so they should just work). If `generateRunCommand` accepts an `environment` parameter, ensure it still works correctly with the new builder.

    Add launch logging:
    5. In `constructWineEnvironment`, after resolve(), log a safe summary using the existing `envLogger`:
       - Bottle name (from bottle.settings.name)
       - Which layers were active (from provenance.activeLayers)
       - Whitelisted non-sensitive key values: DXVK_ASYNC, DXVK_HUD, WINEESYNC, WINEMSYNC, D3DM_FORCE_D3D11, MTL_HUD_ENABLED (only log if present)
       - Do NOT log full environment dict, WINEPREFIX paths, or user-set custom env vars
       - Log level: info (not debug -- this is useful operational data)

       - Test that DLL overrides compose correctly: bottle DXVK on + program custom DLL override -> both present in WINEDLLOVERRIDES
       - Test that constructWineServerEnvironment is removed (compilation check)

    IMPORTANT: Preserve all existing public API signatures where possible. The `environment: [String: String]` parameter on `runProgram` should remain for backward compatibility; `programOverrides` is an additive optional parameter.
