  <action>
    **AudioTroubleshootingWizardView.swift**: The symptom-driven troubleshooting wizard.
    - Input: `@ObservedObject var engine: AudioTroubleshootingEngine`, `onDismiss: () -> Void`
    - Present as a sheet (NavigationStack with title "Audio Troubleshooting").
    - Frame: `.frame(minWidth: 500, minHeight: 400)`

    Wizard flow maps to engine.wizardState:

    `.pickSymptom`:
    - Header: "What audio problem are you experiencing?"
    - Grid of symptom cards (VStack or List). Each card shows AudioSymptom.sfSymbol, displayName, description.
    - Tapping a card calls engine.selectSymptom(symptom).

    `.runningProbes`:
    - ProgressView with "Running audio diagnostics\u{2026}"
    - List of probe names with status indicators (checkmark for complete, spinner for running)

    `.showingFindings`:
    - Header: "Diagnostic Results"
    - AudioStatusView showing current status
    - If findings are empty: "No issues detected. Your audio configuration looks healthy."
    - If findings exist: AudioFindingsView with findings
    - Button row: "Try a Fix" (calls engine.offerNextFix()), "Close" (dismiss)

    `.offeringFix(fixDescription, actionId)`:
    - Header: "Suggested Fix"
    - Fix description text (what will change, how to undo)
    - "Apply Fix" button (primary) -- calls engine.applyFix(actionId:beforeValue:afterValue:). For settings changes, read current value as beforeValue, apply the change, and record afterValue.
    - "Skip This Fix" button (secondary) -- calls engine.offerNextFix() to try next

    `.askingDidItWork`:
    - Header: "Did that fix the audio problem?"
    - "Yes, it's working now!" button (primary, green tint) -- calls engine.userReportsFixed()
    - "No, still having issues" button (secondary) -- calls engine.userReportsNotFixed()
    - "Test Again" button (tertiary) -- runs probes before answering

    `.resolved`:
    - Checkmark circle icon (large, green)
    - "Audio problem resolved!"
    - Summary of what was changed (list attemptedFixes)
    - "Done" button dismisses

    `.escalation`:
    - exclamationmark.triangle icon
    - "We've tried the most common fixes."
    - "Here's what you can try next:" followed by:
      1. "Open Advanced Audio Settings" button (dismisses wizard, sets advancedMode = true)
      2. "Export Audio Diagnostics" button (collects probe results + device info + attempted fixes into a diagnostic string, copies to clipboard)
      3. "Open macOS Sound Settings" button (NSWorkspace to open System Settings > Sound)
    - "Close" button

    **AudioDeviceListView.swift**: Full device list for Advanced diagnostics.
    - Input: `devices: [AudioDeviceInfo]`
    - Layout: List of device rows. Each row shows: device name (bold), transport type badge, sample rate, channel count, "Default" badge if isDefault.
    - Sort: default device first, then alphabetical.
    - If empty: "No output devices detected"

    **AudioDeviceHistoryView.swift**: Device change event log.
    - Input: `history: AudioDeviceHistory`
    - Layout: List of events in reverse chronological order (newest first). Each row shows:
      - Event type icon (speaker.wave.2 for default changed, speaker.slash for disconnected, speaker.wave.3 for reconnected, waveform for sample rate changed)
      - Event description (e.g., "Default output changed to MacBook Pro Speakers")
      - Transport type badge
      - Timestamp (Text(date, style: .relative))
    - If no events: "No device changes recorded this session"
    - "Clear History" button at bottom

    **AudioConfigSection.swift modifications**:
    1. Replace the placeholder troubleshooting sheet with the full AudioTroubleshootingWizardView.
    2. Add AudioDeviceHistory as @State (session-scoped, starts empty).
    3. Wire device change events from monitor.startListening into history.append().
    4. In Advanced mode, add collapsible sections:
       - "All Output Devices" with AudioDeviceListView showing monitor.allOutputDevices()
       - "Device Change History" with AudioDeviceHistoryView showing history
    5. Debounce device change events: use a 2-3 second timer before updating audioStatus for Bluetooth connections (per research pitfall #4). After the debounce, re-query monitor.defaultOutputDevice() and update status.

    **Xcode project**: Add 3 new .swift files to Whisky target.

    **WhiskyApp.swift -- Device change alerts**:
    1. Add an app-level AudioDeviceMonitor as @StateObject (lives for app lifetime).
    2. In the app's main WindowGroup or body, register for device changes using monitor.startListening.
    3. When a device change is received (particularly disconnection or sample rate drop to < 22050 Hz):
       - If Whisky is in foreground: show a non-blocking toast notification. Use the existing toast pattern (Notification.Name extension + overlay). Toast message: "Audio device disconnected: {deviceName}. Switch macOS output device and retest." with info-level styling. Include "Open Audio Settings" action in the toast.
       - If Whisky is backgrounded: attempt macOS notification via UNUserNotificationCenter. Request notification authorization on first use. Fall back to logging if notification fails. Per decisions: best-effort, fallback to logging.
    4. Rate-limit: Track last alert per device name. Don't repeat for same device within 3 minutes (per decisions: "one alert per device per few minutes").
    5. When a previously-disconnected device reconnects: show a quiet "Reconnected" toast (per decisions).
    6. For Bluetooth HFP/low sample rate while bottles are running: check if any bottles have running processes (via ProcessRegistry). If so and sample rate drops to < 22050 Hz, show one non-blocking warning linking to Audio panel (per decisions on proactive exception).
