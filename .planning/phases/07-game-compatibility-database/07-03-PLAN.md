**GameConfigApplicator.swift:**
Create `GameConfigApplicator` as a caseless enum:

**Apply method:**
```swift
@MainActor
public static func apply(
    entry: GameDBEntry,
    variant: GameConfigVariant,
    to bottle: Bottle,
    programURL: URL? = nil
) throws -> GameConfigSnapshot
```

Steps:
1. **Snapshot current state** -- encode `bottle.settings` via PropertyListEncoder into Data. If programURL is provided, also encode the corresponding program's ProgramSettings.
2. Create `GameConfigSnapshot` with bottleSettingsData, programSettingsData, appliedEntryId, appliedVariantId, timestamp.
3. **Apply variant settings to bottle** -- for each non-nil field in `variant.settings`:
   - `graphicsBackend` -> `bottle.settings.graphicsBackend = value`
   - `dxvk` -> `bottle.settings.dxvk = value`
   - `dxvkAsync` -> `bottle.settings.dxvkAsync = value`
   - `enhancedSync` -> `bottle.settings.enhancedSync = value`
   - `forceD3D11` -> set via graphicsConfig or direct property
   - Additional fields as available on BottleSettings
4. **Apply DLL overrides** -- append `variant.dllOverrides` to `bottle.settings.dllOverrides` (deduplicate by DLL name, prefer variant value).
5. **Apply custom environment variables** -- store `variant.environmentVariables` in `bottle.settings.customEnvironmentVariables` (a new [String: String] dict on BottleSettings for bottleUser layer env vars, or use existing mechanism).
6. **Save snapshot** to bottle URL.
7. Return the snapshot.

Note: Winetricks verb installation is NOT done here -- the UI handles that separately per user decision (explicit "Install Required Components..." action). The applicator just records which verbs are needed so the UI can display them.

**Revert method:**
```swift
@MainActor
public static func revert(
    bottle: Bottle,
    snapshot: GameConfigSnapshot
) throws -> [String]
```

Steps:
1. Decode `snapshot.bottleSettingsData` via PropertyListDecoder back to BottleSettings.
2. Replace `bottle.settings` with the decoded settings. (BottleSettings.didSet auto-saves.)
3. If programSettingsData exists, restore each program's settings similarly.
4. Delete the snapshot file from the bottle directory.
5. Return `snapshot.installedVerbs ?? []` as the list of non-reversible components (for UI to display "Settings reverted; installed components remain").

**Pending winetricks verbs method:**
```swift
public static func pendingWinetricksVerbs(
    variant: GameConfigVariant,
    installedVerbs: Set<String>
) -> [String]
```
Returns verbs in `variant.winetricksVerbs` that are NOT in `installedVerbs`.

**Preview diff method:**
```swift
@MainActor
public static func previewChanges(
    variant: GameConfigVariant,
    bottle: Bottle
) -> [ConfigChange]
```

`ConfigChange` struct (Sendable): category (String, e.g., "Graphics", "Performance"), settingName (String), currentValue (String), newValue (String), isHighImpact (Bool).

Iterate variant settings fields. For each non-nil field, compare against current bottle.settings value. Return array of changes grouped by category. Mark graphics backend changes and DLL override additions as high-impact.

7. `testPreviewChangesShowsDiff` -- variant changes graphicsBackend, preview returns ConfigChange with correct before/after

Note: Tests will need to create a temporary bottle directory and minimal BottleSettings for testing. Use the existing BottleTests pattern for creating test fixtures.

**StalenessChecker.swift:**
Create `StalenessChecker` as a caseless enum:

**StalenessReason enum** (String, Sendable): `dateExpired`, `macOSVersionMismatch`, `wineVersionMismatch`

**StalenessResult struct** (Sendable): isStale (Bool), reasons ([StalenessReason]), warningMessage (String?)

**Public API:**
```swift
public static func check(
    testedWith: TestedWith,
    currentMacOSVersion: String? = nil,
    currentWineVersion: String? = nil
) -> StalenessResult
```

Logic (matching CONTEXT.md specification exactly):
1. **Date staleness:** If `testedWith.lastTestedAt` is more than 90 days ago, add `.dateExpired`
2. **macOS version mismatch:** Parse `testedWith.macOSVersion` and current macOS version (default: from ProcessInfo). If user's macOS is newer by >1 minor release (e.g., tested on 15.1, user on 15.3+), add `.macOSVersionMismatch`
3. **Wine version mismatch:** Compare major version numbers. If Wine major version differs (e.g., tested with "9.x", user has "10.x"), unless a compatible range is declared, add `.wineVersionMismatch`

If any reasons exist, `isStale = true`. Generate `warningMessage` combining all reasons into a user-friendly string like "This config was last tested 120 days ago on macOS 15.1 with Wine 9.0."

For macOS version comparison, use `OperatingSystemVersion` from ProcessInfo and parse the string format "15.3.0" manually. For Wine version comparison, compare integer prefix before first dot.

**Tests (add to GameApplicatorTests.swift):**
8. `testStalenessCheckFresh` -- entry tested 30 days ago on same macOS/Wine -> not stale
9. `testStalenessCheckDateExpired` -- entry tested 100 days ago -> stale with dateExpired reason
10. `testStalenessCheckMacOSMismatch` -- entry tested on "14.0.0", current "15.3.0" -> stale with macOSVersionMismatch
11. `testStalenessCheckWineVersionMismatch` -- entry tested with "9.0", current "10.0" -> stale with wineVersionMismatch
12. `testStalenessCheckMultipleReasons` -- entry that triggers both date and version staleness -> both reasons present
13. `testStalenessCheckSameMinorVersionNotStale` -- entry tested on "15.2.0", current "15.3.0" -> not stale (within 1 minor)
