    Add two new optional fields to ProgramSettings (backward-compatible via decodeIfPresent):
    - `activeWineDebugPreset: WineDebugPreset?` -- nil means normal (default), non-nil means user opted into enhanced logging. Transient: set before a diagnostic re-run, cleared after.
    - `lastLogFileURL: URL?` -- path to the most recent Wine log file for this program. Updated after each run.
    - `lastDiagnosisDate: Date?` -- timestamp of the most recent diagnosis run.
    Add these to CodingKeys and encode/decode using encodeIfPresent/decodeIfPresent pattern matching existing overrides field.

    **WineEnvironment.swift additions:**
    In `constructWineEnvironment(for:environment:programOverrides:)`:
    1. After the existing WINEDEBUG=fixme-all line in the wineDefaults layer, add a check:
       If the program's settings has an activeWineDebugPreset that is NOT .normal, inject it via the featureRuntime layer:
       ```swift
       if let preset = programSettings?.activeWineDebugPreset, preset != .normal {
           builder.set("WINEDEBUG", preset.winedebugValue, layer: .featureRuntime)
       }
       ```
       This requires passing ProgramSettings into constructWineEnvironment. Add an optional `programSettings: ProgramSettings? = nil` parameter (backward-compatible).
    2. The featureRuntime layer (priority 7) is above programUser (6), so it correctly overrides any user-set WINEDEBUG for the diagnostic session.
    3. After the run completes, the preset should NOT persist permanently -- the app layer (plan 05-04/05) will set it before a diagnostic re-run and clear it after.

    Do NOT change the default WINEDEBUG value (fixme-all). The preset is opt-in only.

    **Wine.swift modifications:**
    Add a new public static method for post-run classification:
    ```swift
    public static func classifyLastRun(
        logFileURL: URL,
        exitCode: Int32,
        classifier: CrashClassifier? = nil
    ) async -> CrashDiagnosis?
    ```
    Steps:
    1. Read log file content (use StabilityDiagnostics.tailOfLogFile pattern but with larger bound -- 500 lines / 256 KiB for classification)
    2. If no classifier provided, create one with PatternLoader.loadDefaults()
    3. Run classification on background task (Task.detached priority: .utility)
    4. Return CrashDiagnosis (or nil if log unreadable)

    In runProgram(), the existing method returns AsyncStream<ProcessOutput>. The caller already processes .terminated(exitCode). Do NOT modify runProgram's return type. Instead, the classification trigger will be in the app layer (Program+Extensions.swift).

    **Program+Extensions.swift modifications:**
    In launchWithUserMode() (the primary launch path that already handles program execution):
    1. After the for-await loop that processes ProcessOutput, capture the exit code from .terminated(code)
    2. After stream completion, update program.settings.lastLogFileURL to the log file URL (from Wine.makeFileHandle)
    3. If exitCode != 0, OR if end-of-stream .error messages contain crash signatures (simple heuristic: check if any .error line contains "Unhandled exception", "page fault", "device lost"), call Wine.classifyLastRun() and store the result:
       - Persist a DiagnosisHistoryEntry to the program's diagnosis history sidecar
       - Post a Notification (e.g., Notification.Name.crashDiagnosisAvailable) with the CrashDiagnosis in userInfo so the UI can react
    4. The log file URL is available from Wine.makeFileHandle() -- this already creates the log file. Pass the URL through by capturing it when the file handle is created. If makeFileHandle isn't accessible from this scope, store the log URL on ProgramSettings after the handle is created in runProgram.

    **Log file association:**
    In Wine.runProgram(), the log file is created via makeFileHandle(). Add the log file URL to the AsyncStream metadata by:
    - Adding a new ProcessOutput case: `.logFile(URL)` -- emitted once after file handle creation, before .started
    - OR: storing the log URL in a method-scoped variable and exposing it after stream setup

    Choose the approach that minimizes changes to existing callers. If adding a new ProcessOutput case, ensure all existing switch statements handle it (add a default case or explicit handling).

    The simpler approach: After Wine.runProgram creates the file handle, have Program+Extensions capture the log URL from Wine.logsFolder by finding the most recent .log file. This avoids changing ProcessOutput.

    **Notification for UI:**
    Add `Notification.Name.crashDiagnosisAvailable` extension (in Program+Extensions or a small extension file). UserInfo contains:
    - "diagnosis": CrashDiagnosis
    - "programPath": String (program URL path)
    - "logFileURL": URL

    This notification allows the UI layer (Plan 05-04) to present the diagnostics view when a crash is detected.

    Important: Run classification on a background task, not the main thread. The classifier is Sendable. Use `Task.detached(priority: .utility)` to avoid blocking UI.
  </action>
