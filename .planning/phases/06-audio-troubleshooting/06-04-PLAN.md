  <action>
    Create the Audio/ directory under Whisky/Views/.

    **AudioStatusView.swift**: A SwiftUI view showing the audio health status line.
    - Input: `audioStatus: AudioStatus`, `lastTestedDate: Date?`, `defaultDeviceName: String?`, `transportType: AudioTransportType?`, `sampleRate: Double?`, `channelCount: Int?`
    - Layout: HStack with status icon (Image(systemName: audioStatus.sfSymbol) colored by status), status text (audioStatus.displayName), and primary issue description if degraded/broken.
    - Below status: baseline info row showing device name, transport type badge (e.g., "Bluetooth" in capsule), sample rate formatted as "48000 Hz", channel count as "2ch". Show "Last tested" timestamp using Text(date, style: .relative) if available.
    - Use `.font(.caption)` and `.foregroundStyle(.secondary)` for baseline info row.

    **AudioTestButtonsView.swift**: Test button row for Wine audio testing.
    - Input: `@ObservedObject var bottle: Bottle`, `onStatusUpdate: (AudioStatus) -> Void`, `onTestComplete: ([AudioProbeResult]) -> Void`, `testExeURL: URL?`
    - Layout: HStack with 3 buttons:
      1. "Test Wine Audio" (primary, .borderedProminent) -- runs all probes automatically, calls onStatusUpdate with overall status and onTestComplete with results
      2. "Play Test Tone" (secondary, .bordered) -- runs WineAudioTestProbe with --beep flag, then shows inline "Did you hear the tone?" Yes/No buttons
      3. "Refresh" button (SF Symbol: arrow.clockwise, .bordered) -- re-queries CoreAudio state without running Wine probes
    - State: `@State private var isTestRunning: Bool`, `@State private var isToneRunning: Bool`, `@State private var showToneConfirmation: Bool`
    - Show ProgressView() while tests are running. Disable buttons during test execution.
    - "Did you hear the tone?" confirmation: if Yes, update status to .ok for audio stack. If No, add finding "Test tone not audible" with .high confidence.
    - The view constructs probes internally: CoreAudioDeviceProbe with a fresh AudioDeviceMonitor, WineRegistryAudioProbe with bottle, WineAudioTestProbe with bottle and testExeURL. Run probes in Task.

    **AudioSettingsView.swift**: Settings controls with Simple/Advanced toggle.
    - Input: `@ObservedObject var bottle: Bottle`, `advancedMode: Bool`
    - Simple mode (2 controls max per decision):
      1. Audio Driver picker: Picker("Audio Driver", selection: $bottle.settings.audioDriver) with AudioDriverMode.allCases minus .coreaudio (show only Auto and Disabled in Simple)
      2. Latency Preset picker: Picker("Latency", selection: $bottle.settings.audioLatencyPreset) with .defaultPreset and .stable only (omit .lowLatency in Simple)
    - Advanced mode (all controls):
      1. Audio Driver picker with ALL AudioDriverMode.allCases (Auto, CoreAudio, Disabled)
      2. Latency Preset picker with ALL AudioLatencyPreset.allCases (Default, Low Latency, Stable)
      3. Output Device Mode picker (Follow macOS default / Pin to device). When pinned, show a text field or label for pinnedDeviceName.
      4. "Reset Audio State" destructive button (.bordered, .tint(.red)) with confirmation alert. Action calls Wine.resetAudioState(bottle:) in a Task, shows ProgressView during execution.
    - After any setting change via picker, write to Wine registry asynchronously. Use Task { try? await Wine.setAudioDriver(bottle:driver:) } on .onChange of audioDriver. Similarly for latencyPreset -> Wine.setDirectSoundBuffer. Show brief spinner during writes (per research pitfall #3: registry write timing).

    **AudioFindingsView.swift**: Findings list matching Phase 5 remediation pattern.
    - Input: `findings: [AudioFinding]`, `onApplyFix: ((String) -> Void)?` (action ID callback)
    - Layout: VStack of finding rows. Each row shows:
      - HStack: confidence badge (ConfidenceTier color dot), finding description text
      - DisclosureGroup("Details") for evidence and suggestedAction
      - If suggestedAction is non-nil, show a "Fix" button that calls onApplyFix with the action ID
    - Show max 5 findings per decision. If more, show "Show all findings" button.
    - Style: match RemediationCardView from Phase 5 (confidence badges, risk-based styling).
    - "Technical details" DisclosureGroup at bottom (collapsed by default) showing raw evidence strings from all findings.

    All views must have GPL v3 header, `import SwiftUI`, `import WhiskyKit`.

    **AudioConfigSection.swift**: The main Audio section composing all sub-views.
    - Input: `@ObservedObject var bottle: Bottle`
    - State: `@AppStorage("audioAdvancedMode") private var advancedMode: Bool = false` (persist globally, matching graphicsAdvancedMode pattern)
    - State: `@StateObject private var monitor = AudioDeviceMonitor()` (lives for section lifetime)
    - State: `@State private var audioStatus: AudioStatus = .unknown`
    - State: `@State private var probeResults: [AudioProbeResult] = []`
    - State: `@State private var lastTestedDate: Date?`
    - State: `@State private var showTroubleshootingWizard: Bool = false`

    Layout order (per locked decision):
    ```
    Section("Audio") {
        // 1. Status line
        AudioStatusView(
            audioStatus: audioStatus,
            lastTestedDate: lastTestedDate,
            defaultDeviceName: monitor.defaultOutputDevice()?.name,
            transportType: monitor.defaultOutputDevice()?.transportType,
            sampleRate: monitor.defaultOutputDevice()?.sampleRate,
            channelCount: monitor.defaultOutputDevice()?.outputChannelCount
        )

        // 2. Test buttons row
        AudioTestButtonsView(
            bottle: bottle,
            onStatusUpdate: { audioStatus = $0; lastTestedDate = Date() },
            onTestComplete: { probeResults = $0 },
            testExeURL: Bundle.main.url(forResource: "WhiskyAudioTest", withExtension: "exe")
        )

        // 3. Simple/Advanced toggle
        Picker("", selection: $advancedMode) {
            Text("Simple").tag(false)
            Text("Advanced").tag(true)
        }
        .pickerStyle(.segmented)
        .labelsHidden()

        // 4. Settings
        AudioSettingsView(bottle: bottle, advancedMode: advancedMode)

        // 5. Badge if advanced overrides in Simple mode
        if !advancedMode && hasAdvancedAudioOverrides {
            advancedOverridesBadge
        }

        // 6. Findings (if any from last test)
        if !currentFindings.isEmpty {
            AudioFindingsView(findings: currentFindings, onApplyFix: handleApplyFix)
        }

        // 7. Troubleshooting link
        Button("Audio Troubleshooting\u{2026}") {
            showTroubleshootingWizard = true
        }
    }
    ```

    Computed properties:
    - `hasAdvancedAudioOverrides: Bool` -- true if audioDriver != .auto || latencyPreset != .defaultPreset || outputDeviceMode != .followSystem (any non-default in advanced-only settings)
    - `advancedOverridesBadge: some View` -- HStack with exclamationmark.triangle icon and "Advanced audio settings active" text with "Show" button that sets advancedMode = true
    - `currentFindings: [AudioFinding]` -- aggregated from all probeResults
    - `handleApplyFix(_ actionId: String)` -- handle fix actions (toggle settings, etc.)

    onAppear: Start monitor listening for device changes. Update audioStatus when device changes (debounce per research pitfall #4: 2-3s for Bluetooth).

    Sheet: `.sheet(isPresented: $showTroubleshootingWizard)` presents the troubleshooting wizard (placeholder NavigationStack with Text for now -- full wizard in Plan 06-05).
