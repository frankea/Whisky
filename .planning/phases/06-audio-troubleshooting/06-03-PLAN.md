    **AudioProbeResult.swift**: Struct with properties:
    - `probeId: String` (e.g., "coreaudio-device", "wine-registry", "wine-audio-test")
    - `status: ProbeStatus` enum (ok, warning, error, skipped)
    - `summary: String` (one-line human-readable result)
    - `evidence: [String]` (technical detail lines for collapsed display)
    - `findings: [AudioFinding]` (issues detected by this probe)
    - `timestamp: Date`
    Conform to Sendable, Identifiable (id = probeId + timestamp). ProbeStatus enum: Sendable, Equatable with displayName and sfSymbol.

    **AudioProbe.swift**: Protocol and 3 concrete implementations.

    `public protocol AudioProbe: Sendable` with:
    - `var id: String { get }`
    - `var displayName: String { get }`
    - `func run() async -> AudioProbeResult` (never throws -- errors become ProbeStatus.error results)

    **Concrete probe 1: CoreAudioDeviceProbe**
    - Stores a reference to `AudioDeviceMonitor`
    - `id = "coreaudio-device"`, displayName = "CoreAudio Device Check"
    - `run()`: calls monitor.defaultOutputDevice(). If nil -> error result "No default output device found". If found: builds evidence lines (device name, transport type, sample rate, channels). Generates findings: if transport is .bluetooth, add finding "Bluetooth audio device detected" (confidence: .medium, suggestedAction: "Consider using wired headphones for lower latency"). If sample rate < 44100 (unusual), add finding "Low sample rate detected: {rate} Hz" (confidence: .high if < 22050, .medium otherwise). If channels == 0, add finding "No output channels" (confidence: .high).
    - Note: this probe does NOT require a Wine bottle -- it checks macOS audio state.

    **Concrete probe 2: WineRegistryAudioProbe**
    - Stores `bottle: Bottle` (needs @MainActor for registry access)
    - `id = "wine-registry"`, displayName = "Wine Audio Registry Check"
    - `run()`: @MainActor async. Calls Wine.readAudioDriver(bottle:) and Wine.readDirectSoundBuffer(bottle:). Builds evidence with current driver setting and buffer size. Generates findings: if driver is empty string (disabled), add "Audio is disabled in Wine" (.high). If HelBuflen is very small (< 256), add "Very low audio buffer may cause issues" (.medium).
    - Wrap all Wine calls in do/catch -- if registry read fails, return status .error with description.

    **Concrete probe 3: WineAudioTestProbe**
    - Stores `bottle: Bottle` and `testExeURL: URL?` (path to WhiskyAudioTest.exe in app bundle)
    - `id = "wine-audio-test"`, displayName = "Wine Audio Stack Test"
    - `run()`: If testExeURL is nil or file doesn't exist, return .skipped with "Audio test helper not available". Otherwise, construct the Wine execution with curated WINEDEBUG preset: `WINEDEBUG=+mmdevapi,+dsound,+winmm`. Run the exe via Wine.runWine (passing the exe path as a Windows path). Parse stdout for JSON result line (look for line starting with `{"`). If JSON parses successfully with status "ok", return .ok. If status "error", return .warning with api and code from JSON. If exe exits non-zero or JSON not found, return .error. Include Wine stderr in evidence for troubleshooting.
    - For the `--beep` variant (play test tone): add a separate method `func runWithBeep() async -> AudioProbeResult` that passes "--beep" argument.
    - Note: WineAudioTestProbe is the most complex probe. Design it to be resilient -- never crash, always produce a result.

    **scripts/whisky_audio_test.c**: Create the C source from the research code example. Key points:
    - Use WinMM waveOut API (NOT WASAPI) for maximum Wine compatibility per research
    - Include `<windows.h>`, `<mmsystem.h>`, `<stdio.h>`, `<math.h>`
    - Check argc for `--beep` flag
    - Initialize WAVEFORMATEX: 2 channels, 44100 Hz, 16-bit PCM
    - waveOutOpen with WAVE_MAPPER
    - Generate 100ms buffer (silence for automatic test, 440Hz sine wave for beep mode)
    - waveOutWrite, wait for WHDR_DONE
    - Print JSON result line: `{"status":"ok","api":"waveOut","sampleRate":44100,"channels":2,"beep":false}`
    - On error, print: `{"status":"error","api":"waveOut","code":N}`
    - Clean up: waveOutUnprepareHeader, waveOutClose, free

    **scripts/build_audio_test.sh**: Shell script that:
    - Checks for x86_64-w64-mingw32-gcc availability
    - If not found, prints instructions to `brew install mingw-w64`
    - Compiles: `x86_64-w64-mingw32-gcc -o WhiskyAudioTest.exe whisky_audio_test.c -lwinmm -lm`
    - Copies result to `../Whisky/Resources/WhiskyAudioTest.exe`
    - Make executable: `chmod +x build_audio_test.sh`

    After creating the source, attempt to build the exe:
    1. Run `brew install mingw-w64` if not already installed (skip if brew not available)
    2. Run the build script
    3. If compilation succeeds, add WhiskyAudioTest.exe to Whisky/Resources/
    4. If compilation fails (MinGW not available), the probes still work -- WineAudioTestProbe returns .skipped. Log a note in the plan summary.

    **AudioTroubleshootingEngine.swift**: The state machine for the symptom-driven wizard.

    `@MainActor public final class AudioTroubleshootingEngine: ObservableObject` with:

    Published state:
    - `@Published public var wizardState: WizardState = .pickSymptom`
    - `@Published public var selectedSymptom: AudioSymptom?`
    - `@Published public var probeResults: [AudioProbeResult] = []`
    - `@Published public var currentFindings: [AudioFinding] = []`
    - `@Published public var attemptedFixes: [TroubleshootingFixAttempt] = []`
    - `@Published public var isRunningProbe: Bool = false`

    WizardState enum (nested, public, Equatable):
    - `.pickSymptom` -- initial state, show symptom picker
    - `.runningProbes` -- running diagnostic probes
    - `.showingFindings` -- displaying probe results and findings
    - `.offeringFix(fixDescription: String, actionId: String)` -- presenting a fix card
    - `.askingDidItWork` -- "Did this fix it?" Yes/No
    - `.resolved` -- problem fixed!
    - `.escalation` -- max attempts reached, offer export/advanced settings

    Public methods:
    - `func selectSymptom(_ symptom: AudioSymptom)` -- sets selectedSymptom, transitions to .runningProbes, calls runProbes()
    - `func runProbes()` -- sets isRunningProbe, runs all registered probes, collects results and findings, transitions to .showingFindings
    - `func offerNextFix()` -- examines currentFindings and attemptedFixes to find the next untried fix. If found, transitions to .offeringFix. If all tried or count >= maxFixAttempts (3), transitions to .escalation.
    - `func applyFix(actionId: String, beforeValue: String?, afterValue: String?)` -- records the fix in attemptedFixes, transitions to .askingDidItWork
    - `func userReportsFixed()` -- transitions to .resolved
    - `func userReportsNotFixed()` -- if attemptedFixes.count >= 3, transitions to .escalation. Otherwise, calls runProbes() to re-evaluate, then offerNextFix().
    - `func reset()` -- clears all state, returns to .pickSymptom

    Stored properties:
    - `private let probes: [any AudioProbe]` -- injected at init for testability
    - `private let maxFixAttempts = 3`

    Init: `public init(probes: [any AudioProbe])` -- receives probes for dependency injection. The app layer constructs probes with the appropriate AudioDeviceMonitor, bottle, and test exe URL.

    Fix recommendation logic (in offerNextFix):
    Per symptom, define an ordered list of fix action IDs to try. For example:
    - noSound: ["check-audio-driver", "set-coreaudio-driver", "reset-audio-state"]
    - crackling: ["set-stable-latency", "set-coreaudio-driver", "reset-audio-state"]
    - stutter: ["set-stable-latency", "check-audio-driver", "reset-audio-state"]
    - wrongDevice: ["reset-audio-state", "check-system-settings"]
    - menusOnly: ["set-coreaudio-driver", "check-dsound", "reset-audio-state"]
    Skip action IDs that are already in attemptedFixes. If nothing left to try, escalate.

    This logic can be hardcoded initially (per locked decision: "start hardcoded but design for JSON migration without changing probe/result model"). The fix action IDs and their descriptions/instructions are defined as a static dictionary on the engine for now.

    All types must conform to Sendable where appropriate. Use GPL v3 headers.
