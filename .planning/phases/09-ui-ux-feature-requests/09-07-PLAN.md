<task type="auto">
  <name>Task 1: Enhanced Run subcommand with --follow, --tail-log, and deterministic output</name>
  <files>
    WhiskyCmd/Main.swift
  </files>
  <action>
1. **Deterministic launch output (default behavior):** In the `Run` command's `run()` method, when not using `--command` or `--follow`:
   - Run the program via `Wine.runProgram(at:args:bottle:environment:)`
   - On success (result captured): print exactly `Launched "<exe>" in bottle "<bottleName>".`
   - If run log integration is available (from Plan 05), append log path: `Launched "<exe>" in bottle "<bottleName>". Log: <path>`
   - On failure: print error to stderr, exit with non-zero code via `throw ExitCode(Int32(1))` or use the actual exit code from ProgramRunResult

2. **--follow flag:** Add `@Flag(name: .long, help: "Stream program output to terminal") var follow: Bool = false`
   - When `--follow` is set, instead of the default single-line output:
     - Run Wine.runProgram and iterate the AsyncStream<ProcessOutput> that it returns
     - For each `.stdout(line)`: print to FileHandle.standardOutput
     - For each `.stderr(line)`: print to FileHandle.standardError
     - After stream completes, print "Exited with code X" to stderr
     - Return appropriate exit code
   - Note: This requires accessing the AsyncStream from runProgram. Check if runProgram returns the stream or only ProgramRunResult. If it only returns the result, we may need an alternative approach:
     - Option A: Use `Wine.runWineProcess` which returns `AsyncStream<ProcessOutput>` directly
     - Option B: Add a streaming variant of runProgram
     - Choose whichever integrates cleanly without breaking existing API.

3. **--tail-log flag:** Add `@Flag(name: .long, help: "Follow the Wine log file") var tailLog: Bool = false`
   - When `--tail-log` is set:
     - Launch the program normally (print launch confirmation)
     - Then tail the log file (from ProgramRunResult.logFileURL) using a file monitoring loop
     - Read and print new lines as they appear
     - Exit when the Wine process exits (detected by checking if wineserver is still running)

4. **Proper exit codes:** Ensure all error paths use `throw ExitCode(N)` or `throw ValidationError(...)`. The `--follow` mode should exit with the program's actual exit code. Default mode exits 0 on successful launch, non-zero on failure.

5. **Path handling verification:** The current `Run` command uses `URL(fileURLWithPath: path)` which is correct. Verify that:
   - The path argument is treated as a single string by ArgumentParser (it is, since it's `@Argument`)
   - `Wine.runProgram(at:args:)` passes the URL to Process via argument array, not string interpolation
   - No `.split` or whitespace-based parsing of the path anywhere in the chain
   - Add a comment documenting that paths with spaces work because ArgumentParser handles shell quoting and Wine.runProgram uses URL-based argument passing

6. **Keep --command unchanged** per user decision.
  </action>

  <files>

  <action>
1. **Move core shortcut logic to WhiskyKit (ShortcutCreator.swift):**
   - Create `ShortcutCreator` as a caseless enum (following GPUDetection pattern) in WhiskyKit:
     - `static let infoPlist: String` -- the Info.plist template (moved from ProgramShortcut)
     - `static func createShortcutBundle(at appURL: URL, launchScript: String, name: String) throws` -- creates the .app bundle structure (Contents/MacOS/launch script, Info.plist), sets permissions. Does NOT handle icon (that's AppKit/QuickLook, stays in app target).
     - The launch script is passed as a parameter so both app and CLI can provide it.
   - This avoids pulling AppKit/QuickLook into WhiskyKit.

2. **Update ProgramShortcut.swift (app target):**
   - Call `ShortcutCreator.createShortcutBundle(at:launchScript:name:)` for the bundle creation
   - Keep icon extraction (QLThumbnailGenerator) and `NSWorkspace.shared.setIcon/activateFileViewerSelecting` in the app target
   - The app's `createShortcut` becomes: create bundle via ShortcutCreator, then add icon, then reveal in Finder

3. **Add Shortcut subcommand to WhiskyCmd (Main.swift):**
   - Register `Shortcut.self` in the Whisky command configuration subcommands list
   - Implement per research code example:
     ```
     struct Shortcut: AsyncParsableCommand {
         static let configuration = CommandConfiguration(abstract: "Create a macOS app shortcut for a Windows program.")
         @Argument(help: "Name of the bottle") var bottleName: String
         @Argument(help: "Path to the Windows executable") var exePath: String
         @Option(name: .long, help: "Display name for the shortcut") var name: String?
         @Option(name: .long, help: "Output directory (default: ~/Applications)") var output: String?
         @Flag(name: .long, help: "Overwrite existing shortcut") var overwrite: Bool = false
     }
     ```
   - Implementation:
     - Find bottle by name (same pattern as Run)
     - Create Program from URL(fileURLWithPath: exePath) + bottle
     - Generate launch script via `program.generateTerminalCommand()`
     - Determine output directory: `--output` or `~/Applications/` (user local apps dir)
     - Determine app name: `--name` or program name sans .exe
     - Check if .app already exists: if yes and no `--overwrite`, throw ValidationError
     - If `--overwrite`, remove existing
     - Call `ShortcutCreator.createShortcutBundle(at:launchScript:name:)`
     - Print: `Created <appURL.path>`
     - Exit 0 on success, non-zero on failure

4. **PathHandlingTests.swift:** Write tests for path handling:
   - Test that `URL(fileURLWithPath:)` correctly handles: spaces, parentheses, apostrophes, ampersands, unicode characters
   - Test that `String.esc` (the existing shell escape extension) correctly escapes these characters
   - Test that `Program.generateTerminalCommand()` produces valid shell commands for paths with special chars
   - Test round-trip: create URL from path string, generate terminal command, verify path is properly quoted/escaped
   - Example test paths:
     - `/path/to/My Game (x86)/game.exe`
     - `/path/to/John's Game/game.exe`
     - `/path/to/Tom & Jerry/game.exe`
     - `/path/to/Game [Special]/game.exe`

5. Add localization keys (none needed -- CLI output is not localized).
