    Create the Whisky/Views/Diagnostics/ directory and two view files:

    **DiagnosticsView.swift:**
    Main diagnostics view accepting CrashDiagnosis, Bottle, and Program as parameters.

    Layout (summary-first, per locked decisions):
    1. **Header bar:** Program name + bottle name + timestamp + "Export" button + "Copy Report" button
    2. **Headline diagnosis:** Primary category with SF Symbol, confidence tier label (High/Medium/Low), one-line summary. Use large, prominent styling. Show expandable "Why this diagnosis" line explaining the confidence reasoning.
    3. **Category counts:** Horizontal row of compact pills showing category name + count (e.g., "DLL failures: 3"). Each pill is a Button that filters the log to that category.
    4. **Remediation cards section:** Vertical stack of RemediationCardView instances, sorted by confidence DESC then risk ASC. Cards for high/medium confidence shown directly. Low-confidence cards hidden behind DisclosureGroup "Other things to try" (collapsed by default).
    5. **Log section:** Collapsible DisclosureGroup containing LogViewerView (collapsed by default per locked decisions).

    **Responsive split layout:**
    Use GeometryReader to detect available width. If width >= 700pt (Claude's discretion threshold):
    - HSplitView: Left pane (fixed ~350pt) = header + headline + categories + remediation cards. Right pane = log viewer (always visible in split mode).
    If width < 700pt:
    - Vertical stack: header -> headline -> categories -> cards -> collapsible log.

    **State management:**
    @State for active category filter, search text, log filter mode.
    @State for isLogExpanded (false by default).
    Accept an optional onAction closure for remediation card actions (the parent view handles the actual settings changes).

    **On-demand buttons:**
    "Analyze latest log" / "Analyze last run" buttons shown when diagnosis is nil or empty. These trigger re-classification via Wine.classifyLastRun().

    **RemediationCardView.swift:**
    A card component for a single RemediationAction:
    - Card background with subtle border/shadow (match BackendPickerView card style)
    - Header: Action title + confidence tier badge (colored: green=high, yellow=medium, gray=low)
    - Body: "What will change" one-sentence description
    - Footer: Action button(s) + "Undo" path text + "Applies next launch" note (when appliesNextLaunch is true)

    Action button behavior per locked decisions:
    - actionType == .changeSetting && risk == .low: Direct "Apply" button that calls onAction closure
    - actionType == .changeSetting && risk != .low: "Apply..." button with confirmation alert showing what will change
    - actionType == .installVerb: "Install..." button with confirmation showing winetricks verb
    - actionType == .switchBackend: "Switch to [backend]" button with confirmation
    - actionType == .informational: No action button, just informational text (used for anti-cheat "not supported" messages)

    For anti-cheat (.antiCheatUnsupported category): Show "Not supported on macOS" message with helpful next steps (offline mode suggestion, compatibility notes link, export diagnostics button). No fix buttons per locked decisions.

    Each card includes expandable "Why" line explaining confidence reasoning (which log lines matched).

  <name>Task 2: LogViewerView with NSTextView, gutter markers, and filtering</name>
  <files>
    Whisky/Views/Diagnostics/LogViewerView.swift
  </files>
  <action>
    Create an NSTextView-backed log viewer with inline tagging and filtering:

    **LogViewerView.swift:**
    SwiftUI view wrapping NSTextView via NSViewRepresentable.

    **NSViewRepresentable wrapper:**
    - `LogTextView: NSViewRepresentable` with Coordinator
    - NSTextView configured: isEditable=false, isSelectable=true, font=.monospacedSystemFont(ofSize: 11), usesFindBar=true (built-in Cmd+F support)
    - NSScrollView wrapper for scrolling
    - Large text handling: NSTextView natively virtualizes layout via NSLayoutManager

    **Line tagging (per locked decisions):**
    Minimal inline tagging -- only strong matches get visual treatment:
    - Error lines (err:): subtle red background tint + red circle gutter marker (exclamationmark.circle.fill)
    - Warning lines (warn:): subtle yellow background tint + yellow triangle gutter marker (exclamationmark.triangle.fill)
    - Crash signature lines (page fault, c0000005, device lost): red background + red gutter marker
    - Graphics-tagged lines: subtle orange background tint + orange circle (circle.fill)
    - Do NOT color every Wine "fixme:" line -- keep raw log readable per locked decisions

    Apply styling via NSAttributedString attributes on tagged line ranges only. Use the DiagnosisMatch.lineIndex values to know which lines to tag.

    **Gutter implementation:**
    Use NSTextView's NSRulerView or a custom gutter view. Simpler approach: prepend SF Symbol unicode characters to tagged lines in the attributed string (avoiding the complexity of a custom ruler). Or use a thin custom NSView alongside the scroll view that draws markers at matched line positions.

    Choose the simplest approach that works reliably. A colored dot character (Unicode bullet) at the start of tagged lines is acceptable if NSRulerView proves too complex.

    **Filter buttons (above the text view):**
    SwiftUI toolbar/HStack above the NSViewRepresentable:
    - Segmented picker or button group: "Show all" | "Only tagged" | "Crash-related"
    - Search text field (filters lines containing search text)
    - When a filter is active, the NSTextView shows only matching lines (rebuild attributed string from filtered line set)
    - Category count pills from DiagnosticsView can set the active filter

    **Filter modes:**
    - .all: Show complete log
    - .tagged: Show only lines that matched any pattern (have a DiagnosisMatch)
    - .crashRelated: Show only lines matching .coreCrashFatal or .graphics categories
    - .category(CrashCategory): Show only lines matching a specific category (triggered by category pill clicks)

    **Search:**
    Bind to NSTextView's built-in find bar (usesFindBar=true). Also support a SwiftUI text field for quick in-view filtering.

    **Performance notes:**
    - Apply attributed string styling once when diagnosis/filter changes, not per-scroll
    - For very large logs (>10,000 lines), only tag lines referenced in DiagnosisMatch (O(matches) not O(lines))
    - Use NSTextStorage.beginEditing()/endEditing() for batch attribute changes

    **Scroll to match:**
    Provide a method to scroll to a specific line index (called when user taps a category count pill or a match reference in a remediation card).

    GPL v3 file header.
